Simple API Doc Guide

This guide follows the practical usage path: deploy the service, import OpenAPI documents, then maintain Markdown, models, permissions, sharing, and scheduled synchronization. The default port is 9089, and the default account is admin/12345678.

Docker Quick Start

H2 file storage is used by default. To avoid data loss after container recreation, mount /data.

docker run -d --name simple-api-doc \
  -p 9089:9089 \
  -v apidoc-data:/data \
  fugary/simple-api-doc:latest

After startup, open http://localhost:9089/.

The default account is admin/12345678. Change the password immediately after the first production login.

Release Package

Download the latest archive from GitHub Releases, extract it, enter the bin directory, and run the startup script.

Windows

bin\start.bat

Linux

chmod +x bin/start.sh
bin/start.sh

The release package usually includes the executable Jar, config/application.yml, bin startup scripts, and runtime log directories.

MySQL Config

For long-term team usage, MySQL is recommended. Docker deployments can override default database configuration with environment variables.

docker run -d --name simple-api-doc \
  -p 9089:9089 \
  -e DB_TYPE=mysql \
  -e DB_MYSQL_SERVER=host.docker.internal \
  -e DB_MYSQL_PORT=3306 \
  -e DB_MYSQL_DBNAME=apidoc-db \
  -e DB_USERNAME=root \
  -e DB_PASSWORD=12345678 \
  fugary/simple-api-doc:latest
Variable Description
DB_TYPE Database type. Supports h2 and mysql.
DB_DATA_DIR H2 file database directory. Docker deployments should map it to /data.
DB_MYSQL_SERVER MySQL host address.
DB_MYSQL_PORT MySQL port. Default is 3306.
DB_MYSQL_DBNAME Database name. Default is apidoc-db.
DB_USERNAME / DB_PASSWORD Database username and password.
API_TASK_ENABLED Enable background scheduled tasks (including log and cache cleanup). Default is true.
API_LOG_RETENTION_DAYS Access log retention days. Default is 180. Set <=0 to disable auto cleanup.
API_AI_ENABLED Enable AI data generation capability. Default is true.
API_AI_BASE_URL AI Model Base URL.
API_AI_API_KEY AI Model API Key.
API_AI_MODEL AI Model name.
API_AI_CACHE_RETENTION_DAYS AI sample cache retention days. Default is 180.

Local Source Build

The project contains a Vue3 frontend and a Spring Boot backend. Root scripts build the UI first, copy it into backend static resources, and then run Maven packaging.

# Linux/macOS
./build.sh

# Windows
build.bat

You can also execute the steps manually:

cd simple-api-doc-ui
npm install
npm run build
cd ..
mvn clean install -Dmaven.test.skip=true -Pproduction

Login and Home

Open the service URL to enter the login page. The default account is admin/12345678. After login, the left menu includes API Management, System, and Tools.

  • API Management: API Projects, Share Management, Import Management, Project Groups.
  • System: User Management, Task Management, Log Management, Database Pool, H2 Database.
  • Tools: Icon Management, Code Editor, Diff Editor.

Create or Import Project

After entering API Projects, you can create an empty project or import one from Swagger/OpenAPI.

Select import type

File import and URL import are supported. Files are usually .json or .yaml.

Configure authentication and source

URL import can configure Basic, Token, JWT, and other authentication data to fetch protected OpenAPI documents.

Handle duplicate APIs

When importing into an existing project, choose overwrite, skip, or abort duplicate APIs based on your needs.

Save automatic task

After a URL import succeeds, it can be saved as a scheduled import task for future synchronization.

Project Structure

An API project usually contains basic information, folders, API documents, Markdown documents, data models, environment URLs, and sharing configuration.

Object Purpose
Project The top-level container for a set of API docs, including project code, name, icon, and environment configuration.
Folder Organizes APIs and Markdown documents. Nested trees are supported.
API Document Interface path, method, request params, request body, response body, authentication requirements, and examples.
Markdown Document Supplements business notes, onboarding guides, error codes, flow diagrams, and other non-field content.
Data Model OpenAPI component model, supporting schema field inspection and example generation.

API Document View

The API detail page organizes fields by request and response. It supports path parameters, query parameters, headers, body, status codes, authentication information, examples, and schema tree viewing.

For large schemas, use tree or table views and expand nodes gradually to avoid rendering too many fields at once.

Markdown Documents

Markdown documents are suitable for context outside API fields, such as authentication notes, business flows, error code tables, version changes, and onboarding FAQs.

  • They can be placed in the same document tree as API documents and shared together.
  • Markdown editing and preview are supported, which helps fill business semantics that OpenAPI cannot express.
  • Additional Markdown files can be preserved through extension fields during import.

Data Models and Components

Project components maintain OpenAPI schemas, authentication configuration, and security requirements. The system supports viewing model properties, enums, defaults, formats, patterns, length limits, array items, and references.

When a model is reused by multiple APIs, maintain it as a component first and reference it in request or response bodies to avoid duplicate definitions.

History and Locking

API documents and data models support history lists, diff comparison, and recovery. For important manually adjusted content, use locking to prevent later automatic imports from overwriting it.

Recommended rule: let import tasks maintain generated fields, and keep manual Markdown notes and critical explanations locked or maintained as separate documents.

Project Group Permissions

Project groups can assign different access levels to users, including forbidden, readable, writable, deletable, and full access. Admins can manage group members centrally.

For multi-person collaboration, split project groups by business line or system domain, then assign projects to the corresponding groups.

Document Sharing

Share Management generates external documentation links. A share can limit both access scope and interaction capabilities.

  • Set access password and expiration time.
  • Share all documents or selected documents only.
  • Control whether export and debugging are allowed.
  • Configure copyright, watermark, tree icons, and child node display count.

Share URLs usually follow /share/{shareId}.

Online Debugging

API detail pages and share pages can enable request debugging. Debugging supports environment URLs, authentication methods, headers, query parameters, path parameters, body, and response preview.

Send Mode Use Case
Browser Send Use when the target API supports CORS and the request can be sent directly by the browser.
Server Send Use when the target API does not support CORS or needs to be accessed through server-side proxying.

Export and Code Generation

Projects and shared documents support export, which is useful for offline delivery or archiving. The system also integrates OpenAPI Generator and can generate client or server code by language or framework.

Code generation depends on OpenAPI structure quality. Before generation, check schema references, authentication configuration, and the selected API scope.

Scheduled Import Tasks

URL imports can be saved as project tasks. Later, tasks can automatically fetch source OpenAPI documents and update the project. Tasks support manual trigger, stop, disable, and execution status viewing.

Admins can view running task instances, execution status, and exception information in System Task Management.

Data and Logs

  • The default H2 data directory can be configured through DB_DATA_DIR. Docker deployments should mount it to /data.
  • Logs are written to logs/simple-api-doc by default and can be overridden with LOG_DIR.
  • Admins can view operation logs in Log Management and connection status in Database Pool.
  • The H2 console can be enabled with DB_H2_CONSOLE=true. Use it only in controlled environments.

FAQ

What if imported APIs are too many and hard to read?

Use the directory tree to split business modules, and maintain entry explanations with Markdown documents. Large schemas can be viewed progressively through model trees.

Will scheduled imports overwrite manual changes?

The import strategy decides how duplicate APIs are handled. For content that must not be overwritten, maintain it as separate Markdown or lock key documents.

Can a share page allow external users to debug APIs?

Yes. Enable debugging in share configuration and configure environment URLs. If the target API has CORS restrictions, use server send.

Why did data disappear after recreating a Docker container?

H2 file database data must be persisted through /data. If no volume is mounted, the data is deleted together with the container.
